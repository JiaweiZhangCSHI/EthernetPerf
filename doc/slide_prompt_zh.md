## 必须保留的关键信息
- 当前尚无已确认的 10GbE 丢包事故；目前定义为风险/可见性缺口。
- 当前场景仅为 FPGA 与 recon PC NIC 直连，交换机拓扑不属于当前基线。
- 近期 100G 路线仍以 Ethernet-first 为主。
- 若调优后仍反复不达标，SmartNIC/采集卡是第一升级路径。
- FPGA/自定义专用卡保留给更强确定性吞吐需求场景。
- 当前阶段 Windows 优先，但 Linux 仍是可接受升级平台。
- 在启动 100G 采购与验证前，10G 基线必须先形成可信证据链。

## 需要重点展示的数据
- Phase 1 基线：持续有效吞吐 >= 8 Gbps。
- CPU 门槛：验收窗口内平均 CPU 使用率 <= 30%。
- 10G 验收采用双层门槛：冒烟门槛（运行 >= 60 秒且连续 3 次通过）+ 生命周期可靠性门槛（按 CT 场景假设：10 年使用期、约 300 人次/天、单次扫描约 10 秒原始数据突发）。
- 生命周期统计验收以后期单次扫描成功率置信方法为主：
  - Phase 0.5 后期（技术验证导向，弱于 100G）：p0 = 99.50%，95% 置信度，对应 >= 598 个零失败窗口。
  - 100G 资格（更严格）：p0 = 99.99%，95% 置信度，对应 >= 29956 个零失败窗口。
  - 可选更强发布证据：99% 置信度且 >= 46050 个零失败窗口。
  单日/单周/10000 窗口可作为趋势监控展示，但不能替代置信度资格验收。
- 100G 加权评分：
  - Ethernet：92.00 / 4.60
  - SmartNIC/采集卡：73.00 / 3.65
  - FPGA/专用卡：55.00 / 2.75

## 最终测试标准与方法（必须明确写入）
- PPT 必须明确展示后期资格验收采用的单侧置信零失败样本量公式：
  - n >= ln(1 - C) / ln(p0)，其中 n 为所需零失败窗口数，C 为置信度，p0 为目标单次扫描成功率。
- 必须给出以下固定验收门槛：
  - Phase 0.5 后期门槛：p0 = 99.50%，C = 95%，n >= 598，且 k = 0 失败。
  - 100G 资格门槛：p0 = 99.99%，C = 95%，n >= 29956，且 k = 0 失败。
  - 可选更强 100G 发布证据：C = 99%，n >= 46050，且 k = 0 失败。
- 必须给出以下固定执行方法：
  - 步骤1：先执行冒烟门槛（>= 60 秒且连续 3 次通过）。
  - 步骤2：进入按扫描粒度的窗口记录模式，累计样本窗口数。
  - 步骤3：按零失败规则（k = 0）对照所需 n 进行置信度判定。
  - 步骤4：并行检查 CPU 门槛（验收窗口内平均 CPU <= 30%）。
  - 步骤5：若在达到 n 前出现任一失败，判定为“不通过”，触发故障域定位、调优并全量重测。
  - 步骤6：单日/单周/10000 窗口仅用于趋势监控，不得替代资格验收。

## 必须直接使用的 NIC 关键特性表

| 特性 | 为什么重要 | 对我们方案的影响 | 优先级 |
|---|---|---|---|
| DMA | 以较低 CPU 拷贝开销将报文数据从 NIC 搬运到主机内存 | 对 10G 到 100G 扩展中的持续吞吐和 CPU 余量至关重要 | 高 |
| RSS/多队列 | 将接收处理分散到多个队列和 CPU 核心 | 是避免单核瓶颈、支持并行接收线程的关键 | 高 |
| 中断合并 | 在高 PPS 下合并中断以减少上下文切换开销 | 能改善 CPU 效率，但需要在吞吐和时延之间平衡 | 中 |
| RX/TX 环形缓冲 | 定义突发流量到来时系统在丢包前可吸收的深度 | 直接影响瞬时负载尖峰下的丢包行为 | 高 |
| 校验和卸载 | 将校验和计算/校验卸载到 NIC 硬件 | 降低 UDP 接收路径中的单包 CPU 成本 | 中 |
| RSC/LRO | 合并报文以降低逐包处理压力 | 在包率压力主导而非纯带宽主导时更有价值 | 中 |
| Jumbo 支持 | 在相同有效吞吐下减少包率 | 当端到端链路支持 jumbo MTU 时，是 100G/200G 的强杠杆 | 高 |
| 硬件时间戳 | 在 NIC 层提供精确报文时间信息 | 提升丢包、抖动、乱序诊断及根因定位能力 | 中 |
| SR-IOV | 支持虚拟机直接访问 NIC 资源 | 对当前裸机场景相关性低；若后续引入虚拟化则有用 | 低 |

PPT 中必须直接使用以上 NIC 特性表，并保持优先级与影响描述不被改写。

## 必须直接使用的 10GbE 丢包分析内容

### 当前状态
- 当前尚无已确认的 10GbE 丢包事故。
- 现阶段应将 10GbE 丢包视为风险与可见性缺口，必须通过端到端计数器进行验证。
- 当前验证拓扑仅为 FPGA 与 recon PC NIC 直连。

### 故障域分层
1. FPGA 发送端（发送生成与序列连续性）。
2. 直连链路路径（光模块/线缆路径、CRC、物理稳定性、MTU 一致性）。
3. NIC 与驱动路径（RX 队列深度、drops、ring overflow）。
4. UDP socket 与接收应用（buffer overflow、recv 速率、sequence gap）。

## 必须直接使用的丢包定位检查表

| 阶段 | 计数器或信号 | 若不匹配，优先怀疑位置与动作 |
|---|---|---|
| 发送端 TX | FPGA 已发送报文计数与序列连续性 | 若发送计数或序列不稳定，优先检查 FPGA 发送逻辑与 pacing。 |
| 直连链路 | 两端 CRC、error 指示与链路稳定性 | 若链路错误上升，检查线缆、光模块、NIC 端口状态与直连路径上的 MTU 一致性。 |
| NIC RX | NIC 接收报文数、RX drops、队列溢出计数 | 若链路正常但 RX drops 上升，应优先调驱动、ring、中断参数并核对固件。 |
| UDP/Socket | UDP 接收错误/丢弃计数与 socket buffer 使用情况 | 若 NIC 层正常但 UDP drops 上升，应增大 socket buffer 并降低单包处理时延。 |
| 应用序列 | 应用层 sequence gap 或 reorder 计数 | 若 UDP 层看起来正常但应用有缺口，应分析应用流水线和队列交接瓶颈。 |

## 必须直接使用的 Windows 计数器采集表

| 层级 | 命令（PowerShell） | 关键指标 | 解读 |
|---|---|---|---|
| NIC 链路与驱动 | `Get-NetAdapterStatistics -Name "<NIC_NAME>"` | ReceivedBytes、ReceivedUnicastPackets、ReceivedDiscardedPackets、ReceivedPacketErrors | 若发送端稳定但 discard 或 error 增长，优先怀疑 NIC 或链路路径。 |
| NIC 高级配置 | `Get-NetAdapterAdvancedProperty -Name "<NIC_NAME>"` | RSS、中断合并、receive buffers、jumbo 设置 | 若与期望调优值不一致，说明存在驱动配置漂移。 |
| UDP 协议栈 | `netstat -s -p udp` | Datagrams Received、Receive Errors、No Ports | Receive Errors 上升说明 socket 或内核路径存在压力。 |
| IP 栈丢弃 | `netstat -s -p ip` | Received Header Errors、Received Address Errors、Datagrams Discarded | 指标上升通常意味着在进入 socket 前的 IP 处理路径已发生丢弃。 |
| 按秒基线 | `Get-Counter '\Network Interface(*)\Packets Received Discarded','\Network Interface(*)\Packets Received Errors','\Processor(_Total)\% Processor Time' -SampleInterval 1 -MaxSamples 60` | 60 秒内的 discard/error 趋势与 CPU 趋势 | 最早出现偏离的指标通常对应最可能的丢包故障域与责任方。 |

## 必须直接使用的立即动作
1. 增加 FPGA TX、直连链路状态、NIC RX、UDP socket、应用 sequence 计数器的 1 秒同步快照。
2. 进行分级负载测试（2G -> 4G -> 6G -> 8G），识别最早出现计数器分叉的负载阶段。
3. 在当前直连拓扑下执行 A/B 隔离测试（基线驱动 vs 调优驱动、基线 NIC 参数 vs 调优 NIC 参数），锁定故障域。
4. 定义升级规则：当某一阶段连续多个窗口超过丢包门限时，立即分配责任人并进行定向整改后再复测。
